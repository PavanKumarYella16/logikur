LogiKur: Enterprise Multi-Carrier Shipping Orchestration MiddlewareLogiKur is a high-performance, real-time logistics middleware engine designed to transform standard e-commerce checkout systems into margin-protecting decision gates. In legacy supply chain architectures, shipping fees are bound to rigid, static flat-fee rules. This lack of agility directly erodes merchant profit margins on heavy freight, overcharges local customers, and introduces single-point-of-failure risks during carrier service backlogs.LogiKur resolves this operational bottleneck by intercepting live checkout telemetry—including cumulative dimensional weight, alphanumeric destination postal codes, and strict Service Level Agreements (SLA)—to algorithmically calculate and assign the optimal carrier routing path in real time. By matching contract rate cards, applying localized spatial distance surcharges, and executing graceful degradation failovers, LogiKur ensures that enterprise merchants eliminate manual assignment friction, prevent transport overpayments, and secure structural margin protection at the exact point of sale.🛠️ Architectural Topology & Core Data FlowThe application is engineered as a decoupled, asynchronous distributed system structured across three decoupled layers:[ Event-Driven Storefront UI ] ----( JSON Payload via CORS )----> [ FastAPI Application Gateway ]

Static Ingestion Pipeline (API Layer)For immediate processing of standard shipments, incoming order payloads are evaluated against a tiered matrix of operational weight thresholds:Ultra-Lightweight Class ($\le 2.0\text{ kg}$): Segregates small documents and parcel envelopes directly to Deutsche Post (Warenpost Brief) for Economy orders (€2.95) or DHL Express (Express Document) for Priority paths (€14.50).Standard Parcel Class ($2.0\text{ to }31.5\text{ kg}$): Priority orders map to UPS (Express Saver) using a scaled rate equation ($\text{€18.50} + \text{weight} \times \text{€0.40}$). Economy ground options utilize a trailing-digit check on the target postal code string to perform automated, cross-carrier load balancing:Even Terminating Digit: Routed to DHL Paket (Standard Domestic) ($\text{€5.90} + \text{weight} \times \text{€0.15}$).Odd Terminating Digit: Routed to DPD Ground (Standard Domestic) ($\text{€5.90} + \text{weight} \times \text{€0.15}$).Heavy Courier Class ($31.5\text{ to }70.0\text{ kg}$): Maps to DHL Express (Industrial Priority) for premium SLA constraints ($\text{€35.00} + \text{weight} \times \text{€0.80}$) or GLS Heavy (Oversized Parcel) for economy parameters ($\text{€19.90} + \text{weight} \times \text{€0.50}$).Freight Pallet Class ($> 70.0\text{ kg}$): Routes large-scale commercial cargo to DHL Freight (Premium Cargo LTL) for expedited tracking ($\text{€75.00} + \text{weight} \times \text{€1.20}$) or DB Schenker (Economy Pallet Network) for ground bulk transport ($\text{€45.00} + \text{weight} \times \text{€0.75}$).2. Dynamic Relational Matrix Core (LogiKurEngine)When evaluating carrier metrics via database lookups, the execution engine processes the payload through a four-stage algorithmic optimization cycle:A. Spatial Distance Surcharge ModelingTo prevent blocking network bottlenecks associated with third-party mapping APIs during checkout, the system instantly processes the first two digits of the delivery postal code as an integer routing prefix. It evaluates this string against the origin warehouse location identifier (Berlin hub anchor, Zone 10) to determine an absolute spatial zone distance:$$\text{Zone Distance} = |\text{Warehouse Prefix} - \text{Destination Prefix}|$$An automated distance surcharge of €0.40 per zone unit is immediately generated based on this spatial delta.B. Active Contract FilteringThe database layer isolates available carrier records, filtering out profiles that cannot physically accommodate the parcel's mass or do not match the required active boolean status. The engine screens these results against the requested SLA configuration string to eliminate mismatched carrier types.C. Contractual "Tightest-Fit" AllocationEnterprise courier agreements are structured around precise weight-break points. To protect merchant margins, the engine queries active rate tables (DBRateCard) matching the origin warehouse anchor and structures the dataset in ascending sequence by weight capacity:Pythonrate_card = self.db.query(DBRateCard).filter(

Overweight Penalty Overrides & FallbacksIf a parcel's total mass exceeds the explicit contract limits stored in the database matrix, a fallback clause intercepts the execution loop. It pulls the maximum available contract card tier, calculates the overflowing mass differential, and appends a weight penalty parameter of €1.50 per excess kg directly to the baseline valuation:$$\text{Calculated Base Cost} = \text{Contract Base Cost} + ((\text{Total Weight} - \text{Bracket Limit}) \times 1.50)$$E. Mathematical Cost ComparisonThe engine joins the computed components to compile the complete pricing matrix:$$\text{Final Transport Cost} = \text{Base Cost} + \text{Distance Surcharge}$$The system loops through all qualified carrier models, evaluates the lowest-cost path, and returns a unified JSON payload detailing the assigned_carrier, assigned_service, and calculated cost back to the client storefront.🗄️ Database Normalization & Referential IntegrityTo maintain strict transactional security and prevent relational data corruption during background seeding operations (seed.py), the database schema decouples carrier metadata from active contractual rate entries. +-----------------------------+              +-----------------------------+

Eliminating Explicit Primary Key Injection: The database initialization layer avoids explicit hardcoding of primary key IDs during record generation. This lets the relational system database engine manage row sequences natively, preventing key collision errors.ORM Entity Association: Child records (DBRateCard) connect to parent entities by targeting proxy ORM relationship variables directly (DBRateCard(service=dhl_std, ...)). This design ensures that foreign key relationships are seamlessly resolved across transaction states, preventing runtime data exceptions like Error Code 1452 (Foreign Key Constraint Violation).🛡️ Network Resilience & Fault Tolerant FailoversLogiKur enforces pipeline reliability through an integrated cross-origin fallback framework:Graceful Degradation: If an explicit request for an Express courier route triggers a backend validation error or encounters an unmatched carrier rule, the frontend capture system blocks runtime errors from freezing the customer's checkout page.Automated Fallback Routine: The platform immediately intercepts the error response and runs a background fallback procedure, executing a secondary query to resolve a baseline Economy Ground route configuration instead.
